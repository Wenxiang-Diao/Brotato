# Debuff 风险管理系统框架

---

## 1. 系统目标

Debuff 系统用于把“成长”从单纯变强改造成带有代价的构筑过程。

玩家在局内获取武器、状态、奖励、资源、水晶强化时，可能需要接受负面效果。负面效果会影响玩家的战斗节奏、资源效率、状态构筑、路线选择和生存压力。

系统核心不是让玩家难受，而是让玩家思考：

```text
这个奖励是否值得我承担这个风险？
这个 Debuff 是否能被我的角色、武器或水晶转化为收益？
我是否应该在下一次修正窗口移除它？
```

---

## 2. 核心设计原则

| 原则 | 说明 |
|---|---|
| 可理解 | 玩家必须清楚知道 Debuff 的效果 |
| 可预期 | 接受前应展示主要代价 |
| 可修正 | 绝大多数 Debuff 都应存在移除或压制方式 |
| 可构筑 | 一部分 Debuff 可以被特定 build 利用 |
| 可取舍 | 强力奖励应绑定更高风险 |
| 不锁死 | 不应出现让玩家完全无法操作的负面效果 |
| 不纯惩罚 | Debuff 要制造策略，而不是只降低数值 |

---

## 3. 系统循环

```text
战斗 / 节点 / 抽奖
→ 获得奖励
→ 选择是否接受绑定 Debuff
→ Debuff 改变战斗规则
→ 玩家通过角色 / 武器 / 水晶 / 节点管理风险
→ 每 5 级进入修正窗口
→ 保留、压制、替换或移除 Debuff
→ 构筑进一步成型
```

---

## 4. Debuff 存在形式

Debuff 分为三种存在形式：

### 4.1 显性 Debuff

直接显示在 UI 上，拥有明确图标、名称、说明和层数。

示例：

- 移动速度降低 8%
- 拾取范围降低 20%
- 冻结状态持续时间降低

### 4.2 条件 Debuff

只有在特定条件下生效。

示例：

- 生命低于 30% 时受到伤害提高
- 每次重抽后下一波敌人数量提高
- 使用主动技能后短暂降低移动速度

### 4.3 绑定 Debuff

与某个强力奖励绑定存在，奖励失效或被替换后 Debuff 也会消失。

示例：

- 获得红色武器后，治疗效果降低
- 获得高额暴击率后，受到暴击伤害提高

---

## 5. Debuff 上限规则

MVP 阶段建议：

| 项目 | 规则 |
|---|---|
| 同时持有 Debuff 上限 | 5 个 |
| 诅咒类 Debuff 上限 | 1 个 |
| 同类 Debuff 上限 | 2 个 |
| 单个 Debuff 层数上限 | 3 层 |
| 每 5 级修正窗口 | 必定提供至少 1 个修正选项 |

若玩家达到 Debuff 上限，再获得新 Debuff 时触发“超载风险”：

```text
选择 1：放弃当前奖励
选择 2：替换一个已有 Debuff
选择 3：接受高风险，将随机 Debuff 升 1 层
```

---

## 6. Debuff 危险等级

| 等级 | 描述 | 适用场景 |
|---|---|---|
| 轻度 | 小幅影响战斗或资源 | 蓝 / 紫奖励绑定 |
| 中度 | 明显改变操作或构筑 | 紫 / 粉奖励绑定 |
| 重度 | 强烈改变玩法，需要管理 | 粉 / 红奖励绑定 |
| 诅咒 | 高风险高收益，通常只能保留 1 个 | 红 / 金奖励或特殊事件绑定 |

---

## 7. 与核心系统关系

| 系统 | Debuff 作用 |
|---|---|
| 元素 / 状态 | 改变状态附着、持续、抗性或交互效率 |
| 武器 | 强武器绑定负面限制，形成取舍 |
| 角色 | 风险型角色可利用 Debuff 转化收益 |
| 怪物 | 部分怪物施加临时 Debuff，Boss 施加阶段性压力 |
| 节点 | 风险节点、事件节点、休息节点与 Debuff 强关联 |
| 抽奖 / 奖励 | 高品质奖励绑定更强 Debuff |
| 水晶 | 提供局外风险管理和构筑偏向 |

---

## 8. MVP 实现建议

MVP 阶段先实现：

- 30 个基础 Debuff
- 4 个危险等级
- 5 个持有上限
- 每 5 级修正窗口
- 奖励绑定 Debuff
- 节点移除 / 压制 Debuff
- 角色和水晶少量转化 Debuff

后续版本再加入：

- Debuff 组合反应
- Boss 专属诅咒
- 高难度永久诅咒
- Debuff 图鉴
- Debuff 成就挑战
