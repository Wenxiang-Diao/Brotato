# Debuff 平衡性与 UI / UX 规则

---

## 1. 平衡目标

Debuff 应该让玩家感到“有压力但可处理”。

优秀的 Debuff 应满足：

```text
我不喜欢这个代价
但我知道为什么它存在
我也知道可以怎样处理它
甚至某些情况下我可以利用它
```

---

## 2. 强度控制原则

| 原则 | 说明 |
|---|---|
| 轻度可忽略但可感知 | 不影响基础操作 |
| 中度需要调整策略 | 让玩家改变节点或 build 选择 |
| 重度必须有补偿 | 绑定强奖励或高价值事件 |
| 诅咒必须改变玩法 | 不能只是大幅扣数值 |
| 多重叠加要设软上限 | 避免体验崩坏 |

---

## 3. 不推荐设计

以下类型应避免：

| 类型 | 原因 |
|---|---|
| 完全禁止移动 | 破坏核心操作 |
| 完全禁止攻击 | 破坏战斗循环 |
| 完全禁止升级 | 破坏 Roguelite 成长 |
| 随机立即死亡 | 不公平 |
| 说明不清的隐藏惩罚 | 玩家无法决策 |
| 不可移除且持续恶化 | 容易导致弃局 |
| 直接废掉单一 build | 让玩家前期选择失效 |

---

## 4. 数值建议

MVP 阶段建议使用保守数值。

| 类型 | 轻度 | 中度 | 重度 |
|---|---:|---:|---:|
| 攻击降低 | 3%-6% | 8%-12% | 15%-20% |
| 移速降低 | 3%-5% | 6%-10% | 12%-15% |
| 受伤提高 | 4%-8% | 10%-15% | 18%-25% |
| 资源降低 | 5%-10% | 12%-20% | 25%-35% |
| 冷却增加 | 5%-8% | 10%-15% | 18%-25% |

诅咒类不建议单纯套用数值表，应设计成规则变化。

---

## 5. 叠加规则

| 情况 | 规则 |
|---|---|
| 同名 Debuff | 最多 3 层 |
| 同分类 Debuff | 同时最多 2 个 |
| 操作类重度 Debuff | 同时最多 1 个 |
| 诅咒类 Debuff | MVP 同时最多 1 个 |
| 上限已满 | 触发替换 / 放弃 / 升层选择 |

---

## 6. UI 显示规则

Debuff UI 必须优先清晰。

每个 Debuff 显示：

- 图标
- 名称
- 危险等级颜色
- 层数
- 来源
- 是否绑定奖励
- 是否可移除
- 预计持续时间

颜色建议：

| 等级 | 颜色倾向 |
|---|---|
| 轻度 | 黄 / 橙 |
| 中度 | 橙红 |
| 重度 | 红 |
| 诅咒 | 暗红 / 紫黑 |

---

## 7. 选择界面提示

玩家接受 Debuff 前必须看到：

```text
你将获得：雷纹扩散
效果：感电链跳次数 +1
你将承受：电荷反噬 I
效果：未触发感电连锁时，下一次受到伤害 +6%
可修正：可在 5 级修正窗口降层，或在休息节点压制
当前适配：你的 build 已有 3 个感电强化，该代价可控
```

---

## 8. 新手保护

MVP 中建议开启新手保护：

1. 前 3 次升级不出现绑定 Debuff。
2. 第一个 Debuff 必定为轻度。
3. 前 10 级不出现诅咒。
4. 修正窗口第一次必定提供移除选项。
5. 简单难度下重度 Debuff 只来自玩家主动选择。

---

## 9. 失败体验保护

如果玩家连续失败，可以触发隐性保护：

- 降低重度 Debuff 出现概率
- 提高休息节点出现权重
- 提高无 Debuff 奖励出现率
- 提供一次免费修正

该机制不需要明显告知玩家，但可以作为难度调节使用。

---

## 10. 可玩性检查清单

每个 Debuff 上线前需回答：

1. 玩家能否理解它？
2. 它是否有图标和清晰描述？
3. 它是否有至少一种修正方式？
4. 它是否会直接破坏当前核心操作？
5. 它是否有适合绑定的奖励？
6. 它是否能被至少一个角色或水晶利用？
7. 它是否与已有状态系统冲突？
8. 它是否在高难度下仍然公平？

只有全部通过，才建议进入实现。
