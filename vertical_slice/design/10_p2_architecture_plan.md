# P2 工程结构重构计划

> 启动日期：2026-06-07

## 目标

在不改变 P1 已验证玩法、数值和输入的前提下，把单文件原型重构为可继续扩展和测试的模块化工程。

## 架构原则

1. `Main` 只负责状态机、帧循环和模块编排。
2. JSON 读取和数据索引由数据仓库负责。
3. 奖励候选生成由无场景依赖的服务负责。
4. 伤害、护甲、状态和反应规则由战斗规则模块负责。
5. 玩家、敌人和投射物字典统一由实体工厂创建。
6. 指标初始化、累加和保存由指标记录器负责。
7. 对外保留现有 `Main` 方法作为兼容入口，迁移期间测试不直接依赖模块内部实现。

## 模块

| 模块 | 职责 | 不负责 |
|---|---|---|
| `SliceDataRepository` | JSON 加载、类型校验、ID 索引 | 游戏规则 |
| `RunMetricsRecorder` | 指标结构、计数、JSONL 保存 | UI |
| `RewardService` | 合法奖励池、引导候选、重抽条件 | 应用奖励效果 |
| `CombatRules` | 护甲公式、状态附着、状态查询、反应配置 | 实体遍历 |
| `EntityFactory` | 玩家、敌人、投射物数据结构 | 更新循环 |
| `ProgressionService` | 经验升级、基础属性与 Debuff 重算 | 奖励展示 |
| `TargetingService` | 最近、精准、密集目标与类型查询 | 攻击执行 |
| `CombatEventHub` | 发布伤害、状态和反应事件 | 修改战斗结果 |
| `Main` | 状态机、输入、帧循环、模块协调、临时绘制 | 重复实现纯规则 |

## 迁移顺序

- [x] 数据仓库与错误报告。
- [x] 运行指标记录器。
- [x] 奖励服务。
- [x] 战斗规则模块。
- [x] 实体工厂。
- [x] 架构契约测试与完整回归。

## 本轮不做

- 不把字典实体一次性改成完整 Node 场景树。
- 不加入新玩法、数值或内容。
- 不引入对象池，直到真实性能分析证明分配是瓶颈。
- 不拆分绘制层；正式 `Control` UI 属于后续 UI 阶段。

直接在未分析性能前加入对象池，或在仍使用程序化占位视觉时大规模拆 UI，都会增加复杂度但不能提高当前验证价值。
