# P2 工程结构重构完成报告

> 完成日期：2026-06-07

## 重构结果

原始 `main.gd` 共 1215 行，包含数据、指标、奖励、战斗规则、实体构造、成长、目标查询、状态机和绘制。

P2 完成后，纯规则与基础设施迁移到 `game/scripts/core/`：

- `SliceDataRepository`：JSON 加载、错误报告、ID 索引和 schema 清单。
- `RunMetricsRecorder`：指标 schema、反应计数、结算和 JSONL 保存。
- `RewardService`：奖励过滤、保底、风险候选与重抽。
- `CombatRules`：护甲/护盾伤害结果、状态规则和反应查询。
- `EntityFactory`：玩家、敌人和投射物的统一结构。
- `ProgressionService`：经验升级和属性重算。
- `TargetingService`：武器与关卡共用目标查询。
- `CombatEventHub`：伤害、状态和反应事件。

`Main` 现在负责：

- 输入和游戏状态机。
- 每帧更新顺序。
- 调用核心模块。
- 实体集合遍历与场景副作用。
- 当前程序化占位绘制。

## 兼容性

- P1 的输入、数值、固定随机种子、奖励规则和六层流程未改变。
- 保留原有 `Main` 方法作为兼容入口，运行时测试无需绕过正式流程。
- 指标增加 `schema_version=1`，旧核心字段保持不变。
- 数据增加 `manifest.json`，现有 7 份配置内容不变。

## 工程保障

- Godot 模块测试独立验证 8 个核心模块。
- Godot 运行时冒烟测试验证两种模式、战斗、反应、奖励、结算与实体上限。
- Python 工程契约验证模块集合和编排器规模。
- `F9` 调试面板显示 FPS 与关键实体数量。
- Godot 与 Python 同时验证数据 schema 和必需文件。

## 有意不实施

### 对象池

当前投射物上限为 100，且尚无 profiler 证据表明字典分配或释放是帧时间瓶颈。提前加入对象池会增加生命周期和脏字段风险。先使用 `F9` 与 Godot profiler 采样；只有出现分配抖动后再实现。

### 完整 Node 实体化

当前实体仍使用统一工厂生成的字典。直接改为玩家、敌人、投射物场景会同时影响更新、绘制、测试和性能模型，风险超过本轮收益。正式动画、碰撞和美术资产进入后，再按真实节点需求迁移。

### UI 模块拆分

当前 UI 是程序化灰盒绘制。正式 UI 阶段应使用 `Control` 场景和主题资源重建，而不是把临时代码机械移动到另一个脚本。

这些项目不是永久取消，而是设置了明确的触发条件，避免无验证收益的重构。

## 结论

P2 已完成当前灰盒垂直切片所需的工程结构重构。核心规则、数据、指标、成长、目标查询和实体构造已经从主编排器分离，自动测试能够保护模块边界。下一阶段可以在真实性能采样后进入战斗手感与正式 UI，而无需继续扩充基础架构。
