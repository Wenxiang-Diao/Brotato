# P4 阶段 1：正式 UI/UX 现状审计

## 1. 审计范围

本阶段只审计，不改动现有 UI、玩法、输入或数值。

核对内容：

- `main.gd` 中菜单、教程、HUD、升级、修正、暂停和结算状态。
- `project.godot` 的窗口、拉伸与 InputMap 配置。
- 完整设计中的 UI 制作方式、资源格式、图标尺寸和武器信息要求。
- Godot 4.6.3 在 1280×720 下的七个关键界面实际渲染结果。
- 同类型游戏常用的信息分层、升级选择和多输入导航模式。

## 2. 当前实现结论

当前实现已经覆盖完整垂直切片流程所需的信息，但仍属于功能验证 UI，不是可扩展的正式 UI。

| 界面 | 已有内容 | 当前主要缺口 |
| --- | --- | --- |
| 主菜单 | 标题、标准/风险模式、基础操作、固定种子 | 版本文案仍为 P1；无按钮、焦点、鼠标、手柄和设置入口 |
| 新手说明 | 移动、自动攻击、主动技能、反应和目标 | 单页静态文本；无输入设备自适应与分步引导 |
| 战斗 HUD | HP、XP、等级、层数、计时、技能、武器、Debuff、反应说明 | 信息密集且全为文字；XP 无明确标签；武器与状态不可快速识别 |
| 升级选择 | 三选一、类型、说明、风险代价、重抽 | 无图标、品质、焦点态、比较信息和构筑关联提示 |
| 五级修正 | 三项修正方案 | 只有文字列表；风险后果和当前 Debuff 缺少上下文 |
| 暂停 | 继续、返回模式选择 | 无设置、控制、低动态和确认对话框 |
| 结算 | 层数、等级、伤害、受伤和反应次数 | 无武器/构筑总结、主要收益、失败原因与正式按钮 |
| 调试信息 | F8 低动态、F9 性能、F10 跳层 | 调试提示混入正式 HUD，缺少开发构建边界 |

## 3. 已符合设计的部分

- UI 采用 2D 表现，没有引入不必要的 3D 建模。
- 颜色已用于区分生命、主动技能、风险、状态和三种元素反应。
- 遮罩层能够明确暂停战斗并聚焦升级、修正和结算内容。
- 1280×720 下主要文本没有截断，基础流程可读、可操作。
- 升级卡已经区分普通收益与带 Debuff 的风险收益。
- 低动态反馈已有运行时开关，为后续正式可访问性设置提供了基础。

## 4. 问题优先级

### P0：正式 UI 实施前必须解决

1. **UI 全部由 `Node2D._draw()` 固定坐标绘制。**  
   当前没有 `Control`、锚点、容器、焦点链和主题资源。继续在该结构上制作正式资源会导致分辨率适配、输入导航和组件复用成本持续上升。

2. **界面操作硬编码为键盘按键。**  
   菜单、升级、暂停和结算没有统一 UI action，也没有鼠标点击或手柄导航。正式 UI 必须先建立 `ui_accept`、`ui_cancel`、方向导航、重抽和快捷操作的输入映射。

3. **没有统一视觉 token 与组件状态。**  
   字号、颜色、间距、圆角、边框、遮罩、普通/悬停/按下/禁用/聚焦状态尚未形成规范，无法稳定生成或接入图标。

4. **没有字体、图标和正式主题资源。**  
   当前使用 Godot fallback 字体和程序化图形。完整设计要求 UI 经 Figma 设计、以 PNG/SVG 导出并由 Godot UI 实现。

### P1：正式界面的核心体验问题

5. **HUD 文字密度过高。**  
   左上角同时显示生命、经验、等级、层数、时间、技能、武器和 Debuff；右上角长期展示完整状态图例与调试快捷键。战斗中需要持续读取的信息应保留，教学和图例应移出常驻 HUD。

6. **升级卡缺少决策所需的视觉层级。**  
   应提供图标、品质/类型标签、关键数值变化、风险代价、当前等级或持有状态，以及明确的聚焦/选中反馈。

7. **状态与武器缺少图标化识别。**  
   完整设计明确要求“武器品阶颜色”和“状态图标 + 品阶”。当前长文本会在武器数量、名称长度或本地化扩展后溢出。

8. **缺少正式设置与可访问性入口。**  
   低动态模式只能通过 F8 切换，玩家无法发现。暂停菜单至少需要继续、设置、重新开始/退出，并对破坏进度的操作二次确认。

9. **缺少鼠标和手柄完整路径。**  
   同类游戏的升级选择依赖清晰的大卡片焦点与快速确认；当前数字键方案只能作为快捷键，不能作为唯一交互。

### P2：可在核心组件完成后补齐

10. 主菜单“P1 试玩就绪版”已经过期，应改为产品副标题或构建版本信息。
11. 教程应由单页说明改为首次进入时的短步骤提示，并根据当前输入设备显示键鼠或手柄图标。
12. 结算页应增加武器构筑、Debuff、最高反应、关键表现和再次挑战主按钮。
13. 调试提示只应在开发模式或调试面板开启后出现。
14. 颜色不能独立承担含义；品质、状态和风险还需图标形状、标签或边框共同表达。
15. 需要验证 16:9、16:10、超宽屏和窗口缩放下的安全区、文本换行与最小字号。

## 5. 完整设计约束

- UI 流程：Figma 设计，PNG/SVG 导出，Godot UI 实现。
- 武器图标：透明 PNG，建议 64×64 或 128×128。
- 状态图标：建议 32×32 或 64×64。
- 卡牌图标：建议 128×128。
- UI 按钮：透明 PNG/SVG，必须覆盖默认、悬停、按下、禁用和聚焦状态。
- 武器界面必须表达品阶颜色；状态必须表达图标与品阶。
- 美术方向保持 2D/2.5D，不为 UI 或垂直切片优先制作 3D 模型。

## 6. 阶段 2 的架构输入

下一阶段应先建立规范和线框，不直接替换现有界面：

- `CanvasLayer` 作为 UI 根节点。
- `Control` + Container 负责 HUD、卡片、菜单和遮罩布局。
- 一个统一 `Theme` 与颜色、字号、间距、边框、阴影 token 表。
- 可复用的按钮、卡片、状态图标、武器槽、数值条和确认弹窗组件。
- 键盘、鼠标、手柄共用的焦点与导航规则。
- HUD、全屏页面、模态遮罩和开发调试层分离。
- 1280×720 为基准稿，同时定义缩放、安全区和长文本策略。

## 7. 首批图标清单

阶段 3 生成图标前，建议首批控制在能覆盖垂直切片的最小集合：

- 4 个武器：弹壳手枪、磁暴硬币、霜晶、裂纹板砖。
- 4 个基础状态：标记、感电、冻结、灼烧。
- 3 个元素反应：雷链、破碎、热冲击。
- 1 个主动技能：星辉回路。
- 4 个通用 UI：生命、经验、重抽、暂停/设置。
- 风险模式实际使用的 Debuff 图标按数据表补齐。

所有图标必须先以同一画布、描边、透视和色彩规则生成预览板；用户确认后才能进入组件实现。

## 8. 阶段 1 验收结论

- 当前 UI 足够继续验证战斗，但不适合直接叠加正式美术资源。
- P4 应先完成统一规范与 `Control` 架构，再生成图标，最后接入场景。
- 阶段 1 未修改现有 UI、输入、玩法或数值。
- 进入阶段 2 前需要用户确认本审计结论。

## 9. 同类产品参考原则

- Brotato：战斗 HUD 保持紧凑，波次间选择强调快速比较和大面积可选目标。
- Halls of Torment：用明确的类别、数值层级和复古视觉语言统一战斗与升级页面。
- Vampire Survivors：菜单选择可由确认键统一操作，并提供可发现的选项入口。

本项目不直接复制其视觉样式，只采用“战斗时少读字、选择时集中比较、多输入路径一致”的交互原则。
