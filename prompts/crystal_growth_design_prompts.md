# 水晶 / 局外成长系统设计 Prompts

以下 prompts 适合在 Codex 中分阶段执行，用于生成或审查《星链回响》的水晶 / 局外成长系统。

---

## Prompt 1：读取已有设定并分析约束

```text
这是一个本地 Git 仓库中的游戏策划文档项目。请先读取当前仓库中的所有已有文件，尤其是：

1. 项目总策划案 / GDD
2. 元素 / 状态系统设计
3. 武器设计文档
4. 角色设计文档
5. 怪物设计文档
6. 节点 / 关卡路线系统
7. 抽奖 / 奖励系统
8. Debuff 风险管理系统

本阶段只允许阅读和分析，不要创建、修改或删除任何文件。

请完成：
1. 总结当前项目中“古树水晶 / 局外成长”的已有定位。
2. 总结水晶系统必须服务的核心系统：元素、状态、抽奖、Debuff、角色、节点、奖励。
3. 参考 Roguelite、Action Survival、Brotato-like、Hades、Vampire Survivors、Dead Cells 等游戏的局外成长设计经验，分析哪些设计适合当前项目，哪些不适合。
4. 提炼水晶系统的设计原则：不能过于单调、不能破坏局内随机性、不能让数值成长压过机制构筑。
5. 判断当前仓库是否已有 crystals/ 或 meta_progression/ 文件夹。
6. 输出后续设计水晶系统时必须遵守的约束。

注意：不要修改文件，只做分析。
```

---

## Prompt 2：设计水晶系统框架

```text
基于当前项目设定，请设计《星链回响》的“水晶 / 局外成长系统”框架。

本阶段先不要写入文件，只输出系统方案供我确认。

设计目标：
1. 水晶系统是局外长期成长核心。
2. 水晶系统需要影响局内构筑方向，但不能直接替代局内选择。
3. 水晶系统需要与元素、状态、抽奖、Debuff、角色、节点、奖励系统联动。
4. 系统要有长期目标，但不能变成单调的“加攻击、加血量”。
5. 系统应支持 MVP 和后续扩展。

请输出：
- 系统定位
- 核心循环
- 水晶资源类型
- 水晶树结构
- 分支分类
- 节点类型
- 解锁与升级规则
- 与抽奖权重的关系
- 与 Debuff 风险的关系
- 与角色羁绊的关系
- 与节点路线选择的关系
- 平衡性限制
- MVP 实现范围
```

## Prompt 3：设计具体水晶分支和节点

```text
请基于确认后的水晶系统框架，设计具体的水晶树分支与节点。

要求：
1. 至少包含 6 个分支：导能、生息、凝流、灼焰、结晶、星核通用。
2. 每个分支设计 8 到 10 个可升级节点。
3. 每个节点需要包含：名称、层级、效果、升级上限、消耗、设计目的、平衡限制。
4. 节点效果应优先体现机制变化、权重调整、构筑方向引导，而不是单纯数值堆叠。
5. 每个分支都要有一个“关键节点”和一个“终点节点”。
6. 不要让玩家通过水晶系统获得无脑强度碾压。
```

## Prompt 4：写入 crystals/ 文件夹

```text
我确认水晶 / 局外成长系统方案可以继续。

请在仓库根目录创建 crystals/ 文件夹，并写入以下 Markdown 文件：

- crystals/README.md
- crystals/crystal_growth_framework.md
- crystals/crystal_tree_branches.md
- crystals/crystal_node_catalog.md
- crystals/resource_and_unlock_rules.md
- crystals/in_run_integration.md
- crystals/character_bond_integration.md
- crystals/balance_and_ux_rules.md
- crystals/mvp_scope.md

要求：
1. 保持中文策划文档风格。
2. 和已有总策划、元素系统、奖励系统、Debuff 系统、节点系统一致。
3. 不要删除或破坏已有文件。
4. 系统要可落地，适合 MVP 开发拆分。
5. 完成后输出新增文件列表和设计摘要。
```

## Prompt 5：自查一致性

```text
请检查 crystals/ 文件夹中的所有 Markdown 文件。

重点检查：
1. 是否与元素、状态、奖励、Debuff、角色、节点系统冲突。
2. 是否存在过强的永久成长。
3. 是否存在过于单调的纯数值节点。
4. 是否支持 MVP。
5. 是否具备长期扩展空间。
6. README 是否能作为总览文档。

如果发现问题，请直接修复，并输出修复摘要。
```
