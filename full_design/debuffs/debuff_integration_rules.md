# Debuff 系统联动规则

---

## 1. 与元素 / 状态系统联动

Debuff 可以影响状态系统，但不应破坏状态构筑。

可影响方向：

| 方向 | 示例 |
|---|---|
| 状态附着 | 降低某类状态附着概率 |
| 状态持续 | 缩短控制类状态持续时间 |
| 状态叠加 | 增加状态层数但提高反噬风险 |
| 状态反应 | 强化反应伤害但增加自身风险 |
| 状态抗性 | 玩家更容易受到某些状态 |

禁止方向：

- 完全禁止所有状态附着
- 完全关闭核心反应
- 让玩家无法触发已有 build

---

## 2. 与武器系统联动

强力武器可以绑定 Debuff 来控制强度。

示例规则：

| 武器方向 | 适合绑定 Debuff |
|---|---|
| 物理近战 | 受击风险、移速限制、近身压力 |
| 物理远程 | 装填迟滞、弹道偏移、视野压力 |
| 法术近战 | 技能冷却、生命消耗、状态反噬 |
| 法术远程 | 资源消耗、施法后摇、状态不稳定 |

武器升阶到红 / 金时，可以出现绑定 Debuff，但需要让玩家明确选择。

---

## 3. 与角色系统联动

角色可以定义 Debuff 的态度。

| 角色类型 | Debuff 交互 |
|---|---|
| 新手均衡型 | 降低首个 Debuff 危险等级 |
| 状态连锁型 | 状态类 Debuff 可增强连锁反应 |
| 风险收益型 | Debuff 数量越多，伤害越高 |
| 控制 / 生存型 | 可压制操作类和环境类 Debuff |

设计要求：

- 角色不能完全免疫 Debuff
- 风险型角色也必须保留失败风险
- 新手角色只能降低风险，不能删除系统

---

## 4. 与怪物系统联动

怪物可以施加临时 Debuff，但应与局内 Debuff 区分。

| 怪物类型 | Debuff 方式 |
|---|---|
| 普通怪 | 低强度临时 Debuff |
| 精英怪 | 条件性中度 Debuff |
| 特殊机制怪 | 场地或资源压力 |
| Boss | 阶段性 Debuff 或 Boss 专属诅咒 |

临时怪物 Debuff 不计入玩家持有上限，但需要有持续时间。

---

## 5. 与节点系统联动

节点是 Debuff 管理的核心外部结构。

| 节点 | Debuff 关系 |
|---|---|
| 普通战斗 | 少量临时 Debuff 来源 |
| 精英节点 | 高奖励 + 中度 Debuff 风险 |
| Boss 节点 | 高品质奖励 + 重度 / 诅咒风险 |
| 事件节点 | Debuff 交换、转化、诅咒 |
| 商店节点 | 花费金币移除或压制 Debuff |
| 休息节点 | 稳定修正 Debuff |
| 风险节点 | 明确收益与代价绑定 |

---

## 6. 与抽奖 / 奖励系统联动

抽奖系统决定 Debuff 的主要来源。

规则建议：

1. 高品质奖励提高 Debuff 绑定概率。
2. 当前 Debuff 数量越多，后续绑定概率越低。
3. 奖励池需要至少保留 1 个低风险选项。
4. 水晶可以改变 Debuff 类型权重。
5. 特定 build 标签会影响绑定 Debuff 倾向。

---

## 7. 与水晶系统联动

古树水晶是局外风险管理系统。

可提供：

| 水晶节点类型 | 效果 |
|---|---|
| 净化节点 | 提高修正窗口质量 |
| 转化节点 | 将特定 Debuff 部分转为收益 |
| 稳定节点 | 降低重度 Debuff 出现概率 |
| 风险节点 | 提高高品质奖励概率，但提高 Debuff 绑定概率 |
| 专精节点 | 让某类 Debuff 更常出现，同时强化对应流派 |

---

## 8. 与 UI / UX 联动

Debuff 信息必须清晰。

UI 显示至少包含：

- 图标
- 名称
- 危险等级颜色
- 层数
- 来源
- 持续时间 / 是否绑定
- 可修正方式
- 与当前 build 的关系提示

示例提示：

```text
「电荷反噬 II」
危险等级：中度
来源：奖励「雷纹扩散」
效果：未触发感电连锁时，下一次受到伤害 +8%
提示：当前角色擅长感电连锁，该 Debuff 可接受。
```

---

## 9. 与难度系统联动

难度越高，Debuff 系统越激进。

| 难度 | 调整 |
|---|---|
| 简单 | 降低绑定概率，修正窗口更强 |
| 普通 | 标准规则 |
| 困难 | 重度 Debuff 更早出现 |
| 高难 | 诅咒出现概率提高，修正成本提高 |
| Endless | Debuff 上限提高，但奖励也提升 |

---

## 10. 兼容性原则

任何 Debuff 必须通过以下检查：

1. 是否有明确描述？
2. 是否能被玩家理解？
3. 是否存在修正方式？
4. 是否会直接废掉当前 build？
5. 是否能和至少一个系统产生互动？
6. 是否能被 UI 清楚展示？
7. 是否适合短局 Roguelite 节奏？
