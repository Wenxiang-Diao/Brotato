# 节点 / 关卡路线系统设计 Prompts

以下 prompts 适用于在 Codex 本地仓库环境中分阶段执行，用于设计《星链回响》的节点式 Roguelite 关卡路线系统。

---

## Prompt 1：读取项目设定，不修改文件

```text
这是一个本地 Git 仓库中的游戏策划文档项目。请先读取当前仓库中的所有已有文件，尤其是：

1. 项目总策划案 / GDD
2. 元素系统 / 状态系统设计文档
3. 武器设计文档与 weapons/ 文件夹，如果存在
4. 角色设计文档与 characters/ 文件夹，如果存在
5. 怪物设计文档与 monsters/ 文件夹，如果存在

本阶段只允许阅读和分析，不要创建、修改或删除任何文件。

请完成以下任务：

1. 总结当前游戏的一局核心循环。
2. 总结当前项目中节点式 Roguelite 结构的已有描述。
3. 分析节点系统需要如何承接战斗、抽奖、debuff、商店、休息、Boss 和局外成长。
4. 分析当前已有角色、武器、怪物内容对关卡路线系统提出了哪些需求。
5. 参考 Slay the Spire、Peglin、Brotato、Vampire Survivors-like、Hades 等 Roguelite / Action Survival 游戏的路线与关卡节奏设计，提炼适合本项目的节点设计原则。
6. 判断当前仓库是否已有 nodes/ 或 stages/ 文件夹。
7. 输出你读取到的关键文件列表和后续设计约束。

注意：
- 不要修改文件。
- 不要创建文件。
- 不要提交 commit。
- 只做理解、分析和约束总结。
```

## Prompt 2：设计节点 / 路线系统框架，不写文件

```text
基于你刚才读取到的项目设定，请先设计一个适用于本项目的“节点 / 关卡路线系统框架”。

本阶段不要创建、修改或删除任何文件，只输出设计框架供我确认。

框架至少包含：

1. 系统目标
   - 为什么本项目需要节点路线系统
   - 节点系统如何服务元素构筑、风险管理和短局 Roguelite 体验

2. 一局路线结构
   - 一局总层数
   - 每层节点数量
   - 分支数量
   - 每几层出现精英 / 商店 / 休息 / Boss
   - 节点之间如何连接

3. 节点类型
   至少包含：
   - 普通战斗节点
   - 精英战斗节点
   - Boss 节点
   - 事件节点
   - 商店节点
   - 抽奖节点
   - 休息节点
   - 宝藏 / 水晶节点
   - 风险节点

4. 节点奖励规则
   - 经验
   - 金币
   - 武器
   - buff
   - debuff
   - 水晶碎片
   - 状态强化
   - 角色羁绊

5. 路线生成规则
   - 节点随机生成规则
   - 强制节点规则
   - 节点权重
   - 避免重复规则
   - 分支策略性

6. 难度递增规则
   - 层数难度
   - 怪物强度
   - 怪物组合
   - 精英词缀
   - Boss 阶段
   - debuff 压力

7. 玩家决策点
   - 为什么玩家选择某条路线
   - 如何让不同 build 选择不同节点
   - 如何避免路线选择变成纯随机

8. MVP 范围
   - 第一版需要实现哪些节点
   - 哪些内容可以后续扩展

设计要求：
- 节点系统必须服务于“元素构筑 + 技能联动 + 节点推进”。
- 节点系统必须和 debuff 风险管理系统强绑定。
- 节点系统必须支持短局节奏，不要过度拉长单局。
- 节点奖励需要推动 build 成型，而不是只给数值。
- 节点风险与收益需要明确。
- 机制要可开发落地。
```

## Prompt 3：正式创建 nodes/ 文档

```text
我确认节点 / 关卡路线系统框架可以继续。

现在请正式在仓库中创建节点 / 关卡路线系统文档。

请执行以下文件操作：

1. 在仓库根目录创建 `nodes/` 文件夹。
2. 创建 `nodes/README.md` 作为节点系统总览。
3. 创建节点路线设计框架文件。
4. 创建节点类型详细设计文件。
5. 创建路线生成规则文件。
6. 创建节点奖励规则文件。
7. 创建难度递增与关卡节奏文件。
8. 创建事件节点示例文件。
9. 创建 Boss 路线与章节结构文件。

建议文件结构：

nodes/
├── README.md
├── node_route_design_framework.md
├── node_types.md
├── route_generation_rules.md
├── node_reward_rules.md
├── difficulty_and_pacing.md
├── event_node_examples.md
└── boss_and_chapter_routes.md

设计要求：

1. 保持中文策划文档风格。
2. 内容要能直接给开发、关卡策划、数值策划继续拆解。
3. 和总策划、元素系统、角色、武器、怪物系统保持一致。
4. 不要删除或破坏已有文件。
5. 节点设计要支持 MVP 阶段，同时保留长期扩展空间。
6. 每种节点都要有明确功能、奖励、风险、适合的玩家决策场景。
7. 路线生成规则要清晰、可实现。
8. 难度递增规则要与怪物、精英、Boss、debuff 压力挂钩。
9. 事件节点示例至少设计 12 个。
10. 输出新增文件列表和设计摘要。
```

## Prompt 4：自查与修复

```text
请检查刚才新增的 `nodes/` 文件夹和其中所有 Markdown 文件。

检查内容包括：

1. 是否存在 `nodes/README.md`。
2. 是否包含节点系统框架、节点类型、路线生成、奖励规则、难度节奏、事件示例、Boss 路线等文档。
3. 各文件中的节点名称和分类是否一致。
4. 节点奖励是否与元素、武器、debuff、水晶、角色羁绊系统挂钩。
5. 路线生成规则是否可实现。
6. 是否存在收益过高但无风险的节点。
7. 是否存在风险过高但无足够奖励的节点。
8. 是否有节点与已有怪物、角色、武器或元素系统冲突。
9. 是否适合作为 MVP 第一版落地。
10. 是否有后续扩展方向。

如果发现问题，请直接修复。

完成后输出：
1. 修复了哪些文件
2. 修复了哪些问题
3. 当前节点系统是否适合作为 MVP 第一版
4. 仍需人工确认的设计风险
```
